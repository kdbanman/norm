07 — How is idle_gap_s accumulated across skipped ticks?

Spec clarification Resolved
Affects: REQ-RECORD-002; concept §8, §10.4

What's unclear

When the user is idle, a tick is skipped and the elapsed idle is "buffered" to stamp onto the next stored capture as idle_gap_s. But if several consecutive idle ticks pass before the next stored capture, it's undefined whether idle_gap_s is:

  • the last observed HIDIdleTime,
  • the sum of idle values across the skipped ticks, or
  • the wall-clock gap between the last stored capture and the next stored one.

Why it matters

REQ-RECORD-002 only asserts idle_gap_s >= observed idle, which all three options satisfy for a single tick. The persistent-loop behavior (REQ-RECORD-005) needs a defined rule, and any exact assertion depends on it.

Options / suspected answer

  • (a) Last observed idle — simplest; loses information across multiple skips.
  • (b) Sum of skipped idle — matches the "buffer idle gap" wording; can overcount if ticks overlap the same continuous idle stretch.
Suspected most meaningful (c) Wall-clock gap — idle_gap_s = ts(next stored) − ts(last stored) − accounted active time. Directly answers "how long was nothing recorded," which is the point of the field for time-accounting use cases. Falls back to (a) if the loop is kept stateless.

Decide, and consider making the >= in REQ-RECORD-002 an exact relation once the rule is fixed.

Resolution

Decided(c) Wall-clock idle gap. idle_gap_s on a stored capture = the total wall-clock seconds skipped due to idle since the previous stored capture (the sum of the contiguous skipped idle ticks' durations), reset once it is stamped. It answers "how long was nothing recorded," which is the point of the field for time accounting. REQ-RECORD-002's >= can be tightened toward equality once the loop cadence is fixed.

Rationale: the user prioritizes accurate time accounting and asked for the exact time away; summing the skipped ticks equals the real idle stretch.
