12 — Are interval reports ever stored? ("not stored by default")

Spec clarification Resolved
Affects: REQ-INTERVAL-001, REQ-INTERVAL-005 (--output); concept §6, §10.11

What's unclear

The data model says interval reports are "written to file or stdout; not stored by default." The phrase "by default" implies there is a way to store them — but no flag or command for that is specified anywhere. REQ-INTERVAL-005 only covers --output <file> (a plaintext file), which is not the same as storing an encrypted row in the index.

Why it matters

Either there's an unspecified --store-style flag that needs its own requirement, or the "by default" wording is misleading and should be removed. Tests shouldn't assume a storage path that doesn't exist.

Options / suspected answer

  • (a) Add --store/--save — writes an encrypted interval-report row to the index (mirroring preprocess), distinct from --output (plaintext file). Would need its own requirement and data model entry.
Suspected simplest (b) Interval reports are never persisted in the store. Drop "by default" from the wording. Output goes to stdout or --output <file> only. Keeps the store model simpler; re-running interval is cheap if preprocess is cached.

Resolution

DecidedInterval reports are always stored (neither option (a) nor (b) — the user chose to persist every report). Drop "by default": every report interval run persists an encrypted interval-report row + markdown blob, and also emits to stdout (or --output <file> as plaintext). New data-model entity interval_report: id, generated_at, range_from, range_to, model, prompt_id, source_preprocess_ids[], markdown_ref. Re-running appends a new row (history retained, keyed by generated_at). Deletable via prune --include reports ([10]).

Rationale: the stored history is the durable standup record; appending keeps a timeline of how summaries evolved across runs/prompts.
