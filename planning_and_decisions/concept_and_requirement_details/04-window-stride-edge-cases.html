04 — Window/stride edge cases (too few captures, trailing remainder)

Spec clarification Resolved
Affects: REQ-PREPROCESS-001, REQ-PREPROCESS-006; concept §9, §10.8

What's unclear

The window count formula floor((N − K) / J) + 1 is consistent across both docs (N=12, K=6, J=3 → 3 windows), but two boundary cases are undefined:

Case 1 — Fewer captures than the window (N < K)

The formula goes to 0 or negative. E.g. N=4, K=6, J=3 → floor(-2/3)+1 = 0. Is this exit 0 with "0 windows," or exit 5 ("not enough captures")? Only N = 0 is specified today (REQ-PREPROCESS-006 → exit 5).

Suspected Succeed with 0 windows, exit 0 for 0 < N < K. It's not an error to lack a full window yet — more captures will accumulate. Reserve exit 5 strictly for 0 captures.

Alternative: treat 0 < N < K as exit 5 too. Pick one explicitly.

Case 2 — Trailing remainder

When (N − K) is not a multiple of J, the last captures fall into no window. Example: N=13, K=6, J=3 → 3 windows covering indices 0–11; capture 12 is never summarized. Drop the tail, or emit a final short/shifted window?

Suspected Drop the tail (consistent with the bare floor formula). The trailing captures will be picked up once enough accumulate to start a new window. This must be documented explicitly so that "coverage" in [01] doesn't treat the dropped tail as perpetually uncovered.

Alternative: emit a final partial window of size < K (changes the count formula and the REQ-PREPROCESS-001 assertion).

Why it matters

These cases determine the exact pass count for REQ-PREPROCESS-001 and interact directly with "coverage" in [01]. An uncovered tail capture could force an extra window on every --auto-preprocess run if not handled explicitly.

Resolution

Decided

  • Case 1 (N < K): N = 0 → exit 5 (NO_CAPTURES); 0 < N < K → error, exit 5 with a distinct code NOT_ENOUGH_CAPTURES ("need ≥K captures; have N"). Reverses this file's original suspicion — the user chose to treat too-few as an error.
  • Case 2 (trailing remainder): drop the tail — the bare floor((N−K)/J)+1 formula stands. Dropped captures are summarized once enough accumulate to start a new window, and are not treated as perpetually uncovered by coverage [01].

Rationale: a distinct error code keeps the numeric exit table stable (still 5) while honoring "treat too-few as an error"; drop-tail keeps the window math simple and idempotent.
