15 — Are output strings contractual, and how strict are assertions?

Spec clarification Resolved
Affects: essentially every requirement with a stderr/stdout text expectation

What's unclear

Both docs quote specific human-readable messages — "idle 180s; skipped", "0 new windows", "store already initialized; use --force", "duplicate; extended C(n)" — but never say whether these are contractual exact strings or just illustrative examples. Requirements paraphrase them loosely ("stderr explains…", "stderr names the missing permission").

Why it matters

This decides how brittle the test suite is. Asserting exact prose means every wording tweak breaks tests; asserting nothing about messages means we can't verify "names the missing permission." We need a policy.

Suggested layered policy

Lean
  1. Exit codes are always contractual — assert exactly.
  2. --json output (where supported) is contractual — assert structure, keys, and values.
  3. Human prose is not exact-matched. Assert stable substrings/keywords that carry the requirement's intent — e.g. message contains "force" for init-clobber; names "Screen Recording" / "Accessibility" for permission errors.
  4. Consider stable message codes or a --json error envelope ({"code": "STORE_LOCKED", "exit": 3, ...}) so negative tests can assert a code instead of prose. This is the most robust option and worth adding to the spec.

Resolution

DecidedLayered contract (all four levels adopted, the most robust option).

  1. Exit codes are contractual — assert exactly.
  2. --json output is contractual — assert structure, keys, values. Failures under --json emit a structured envelope {"error":{"code":"STORE_LOCKED","exit":3,"message":"…"}}.
  3. Stable string error codes back every failure mode (NOT_INITIALIZED, STORE_LOCKED, NO_CAPTURES, NOT_ENOUGH_CAPTURES, COVERAGE_MISSING, MODEL_UNAVAILABLE, INVALID_MODEL_REF, PERMISSION_MISSING, MACAPPTREE_MISSING, UNKNOWN_ID, USAGE_ERROR, …).
  4. Human prose is not exact-matched — assert stable keyword substrings (e.g. contains "Screen Recording"/"Accessibility"; contains "force").

Rationale: the user is mostly-automated, so machine-readable codes/JSON are first-class and tests assert codes, not prose.
