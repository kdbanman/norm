18 — Verifying encryption at rest (without involving norm)

Verification approach Resolved
Affects: REQ-SEC-001, plus the "blobs are ciphertext" clauses in REQ-RECORD-001 and REQ-PREPROCESS-001

The gap

"All on-disk capture data is ciphertext" must be checked by inspecting files directly, with no norm involvement — so it's an executable criterion, just not a norm CLI assertion.

Options (all executable, non-CLI)

Suspected core Magic-byte / parser negative tests — for every blob under data_dir/blobs/:
  • Assert it does not start with the PNG/JPEG magic signature and fails to decode as an image.
  • For AX blobs, assert json.loads() fails.
  • For the index, assert it is not a readable SQLite database — no SQLite format 3\0 header (SQLCipher encrypts the header too).
  • Entropy check — assert blob byte-entropy is near-random (ciphertext), catching accidental plaintext that happens not to match a known magic number.
  • Round-trip control — confirm show --export / export can recover valid PNG/JSON with the key, proving the at-rest files are the encrypted form of real data, not garbage.

Pair "no plaintext at rest" with the transient-decryption check during reporting in [20]. These are setup/inspection helpers run around CLI calls, not assertions on norm's stdout.

Resolution

Decided Adopt the magic-byte / parser negative tests as the core executable criterion, run directly on files with no norm involvement: every blob under data_dir/blobs/ must not begin with a PNG/JPEG signature and must fail to decode as an image; AX blobs must fail json.loads(); data_dir/index.db must not carry the SQLite format 3\0 header; and data_dir/key.json holds only the Argon2id-wrapped key (no usable key material). Add an entropy check (near-random bytes) and a round-trip control (show --export/export recovers valid PNG/JSON with the password) to prove the at-rest files are encrypted real data. Covers REQ-SEC-001 and the ciphertext clauses of REQ-RECORD-001/REQ-PREPROCESS-001.
