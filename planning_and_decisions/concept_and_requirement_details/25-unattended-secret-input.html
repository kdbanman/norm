25 — Non-interactive passphrase input (passwd, daemon)

Verification approach Spec clarification Resolved
Affects: REQ-INIT-003, REQ-SEC-005 (passwd), REQ-RECORD-009 (daemon under a passphrase-wrapped store)

The gap

NORM_PASSPHRASE covers non-interactive init (REQ-INIT-003), but two flows need more:

  • passwd prompts for old + new (new twice). A single NORM_PASSPHRASE can't drive a two-secret rotation, so REQ-SEC-005 can't be tested non-interactively as currently specified.
  • Daemon (record --install) on a passphrase-wrapped store has no specified source for the passphrase at runtime. A launchd agent runs unattended — where does the secret come from?

Options

Testing passwd non-interactively

Suspected Define NORM_OLD_PASSPHRASE + NORM_NEW_PASSPHRASE (or accept old+new on consecutive stdin lines) so rotation is scriptable. Then REQ-SEC-005's "old fails (3) / new unlocks (0)" becomes a clean automated check.

Daemon + passphrase-wrapped store — pick a policy

This is also a spec gap: the daemon's unlock source is unspecified.

  • (a) Disallow passphrase-wrapped stores under launchd — daemon requires the unwrapped Keychain-only key. Simplest and safest.
  • (b) Read NORM_PASSPHRASE from the agent's plist environment — flag the risk: the secret then sits in a plaintext plist, partially defeating the passphrase. If allowed, document the tradeoff explicitly.
  • (c) Use a Keychain-stored wrap the agent can unlock via its ACL, with no passphrase at runtime. Most secure, most complex.

Decide the daemon policy (it is a spec gap, not just a test gap), then add the env seams for hermetic tests.

Resolution

Decided Settled as part of the app-password key model (no Keychain; security model §7). Daemon policy: a passphrase-wrapped store running unattended unlocks from a chmod 400 password file under ~/.norm/ (or NORM_PASSPHRASE), exactly like an unencrypted on-disk SSH/SSL key — option (b) in spirit, but in a dedicated chmod 400 file rather than the launchd plist. This deliberately trades away the same-user guarantee for headless operation and is documented as such. record --install reads that file at runtime (REQ-RECORD-009). passwd: rotation is scriptable via NORM_OLD_PASSPHRASE + NORM_NEW_PASSPHRASE (or old+new on consecutive stdin lines), so REQ-SEC-005's "old fails (3) / new unlocks (0)" is a clean automated check.
