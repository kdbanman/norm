09 — How do commands behave before init (no store at all)?

Spec clarification Resolved
Affects: all data/report/status commands; concept §10 (only init states are enumerated)

What's unclear

The worked examples cover init and init over an existing store, plus "store locked" (§10.20 → exit 3). But the "never initialized" state — no ~/.norm, no data_dir, no index, no Keychain key — is not enumerated for status, list, show, export, prune, record, report, or config get/set.

"Locked" (key absent but store exists) and "absent" (store was never created) are different situations and may warrant different exit codes and messages.

Why it matters

Every command needs a defined behavior from a clean machine. Tests will run against a freshly created home directory with nothing initialized.

Options / suspected answer

Commands that require the store

Suspected best Exit 5 (not_found) with a clear "not initialized; run norm init" message. The store is a missing resource, distinct from a present-but-locked store (exit 3).
  • 3 (auth) — treat "no key/no store" uniformly as not-unlockable. Simpler, but conflates "absent" with "locked."
  • 2 (usage) — least appropriate; this isn't an argument error.

Special case: config

config operates on ~/.norm only, not on the store. Does it work before init?

Suspected config get/set require ~/.norm to exist (written by init). If ~/.norm is absent, they fail with "not initialized." They do not create the config file on demand.

Resolution

Decided

  • Store commands (record, report, list, show, export, prune) on a never-initialized machine → exit 5 with code NOT_INITIALIZED ("run norm init"). Distinct from present-but-locked (exit 3, STORE_LOCKED).
  • config get/set require ~/.norm to exist; absent → exit 5 NOT_INITIALIZED. They do not create the file on demand.
  • status is special: it always exits 0 and reports initialized (bool), locked (bool), counts when unlockable (else null), and daemon state — so automation can probe a clean machine without handling an error.

Rationale: "absent" ≠ "locked" deserves its own code; status as a never-failing probe is friendlier for the user's mostly-automated use.
