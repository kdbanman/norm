05 — --from / --to / --last semantics and interaction

Spec clarification Resolved
Affects: REQ-INTERVAL-001/002, REQ-PREPROCESS-004, REQ-DATA-002/005, REQ-DATA-006 (--before); conventions time_args

What's unclear

The three time flags are used interchangeably in the examples, but their relationships aren't defined. Four sub-questions:

Combination: --last + --from/--to

What happens if both are given? Error, or does one win?

Suspected Make --last mutually exclusive with --from/--to. Combining them is a usage error → exit 2.

Defaults when only one side is given

If only --from is given, is --to implicitly now? If neither is given to report preprocess / list, is the range "all captures"? (report interval requires a range per REQ-INTERVAL-002 — confirm.)

Suspected --to defaults to now when only --from is given. For preprocess and list, omitting all range flags = all captures. report interval with no range → exit 2.

Symmetry: relative offsets on --to

Conventions say --from accepts relative (-24h, yesterday) but --to accepts only ISO-8601 or now. Is --to -1h really invalid while --from -1h is valid?

Suspected Allow relative offsets on both --from and --to. Resolve all relative/now values against a single command-start timestamp. (Revise the conventions doc's stated asymmetry.)

Calendar words (yesterday)

Which timezone and which day boundaries (local? DST edges?)

Suspected Resolve in the local timezone; yesterday = [00:00, 24:00) of the prior local day.

Why it matters

Range parsing underlies nearly every data/report command. Tests need exact rules for defaults, precedence, and which forms are accepted on which flag.

Resolution

Decided

  • --last is mutually exclusive with --from/--to; combining them → exit 2.
  • --to defaults to now when only --from is given. For list and report preprocess, omitting all range flags = all captures. report interval still requires a range (none → exit 2).
  • Relative offsets allowed on both --from and --to (e.g. --to -1h is valid). All relative/now values resolve against a single command-start timestamp. (Revises the conventions doc's stated asymmetry.)
  • Calendar words resolve in the local timezone; yesterday = [00:00, 24:00) of the prior local day. prune --before accepts the same forms.

Rationale: predictable, symmetric range parsing across every data/report command.
