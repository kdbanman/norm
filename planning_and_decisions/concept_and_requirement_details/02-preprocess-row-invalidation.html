02 — What is a preprocess row's identity, and what invalidates it?

Spec clarification Resolved
Affects: REQ-PREPROCESS-003 (idempotent unless --force), REQ-CONFIG-003 (prompt change), REQ-CONFIG-004 (model change); concept §6, §10.9

What's unclear

A preprocess row stores window_start, window_end, capture_ids[], model, prompt_id, markdown_ref. Idempotency "skips covered windows," but it is not stated which fields form the row's identity — i.e., what makes a window count as "already done."

The tension: if identity is just the captures/window, then changing the prompt (REQ-CONFIG-003) or model (REQ-CONFIG-004) would not trigger a recompute on the next run, silently contradicting those requirements. If identity includes model + prompt, then a prompt/model change naturally produces "new" (uncovered) windows.

Why it matters

Determines the pass condition for REQ-PREPROCESS-003 and whether CONFIG-003/004 can be verified by a plain re-run rather than only via --force.

Options / suspected answer

Suspected Identity = (capture_ids, model, prompt_id). A window is "done" only if a row exists matching all three. Changing model or prompt makes prior rows non-matching, so the next run recomputes those windows without --force. --force recomputes even on an exact identity match.

Alternative: identity = (window_start, window_end) only, and rely on --force for prompt/model changes. Rejected — makes CONFIG-003/004 awkward and surprising.

Sub-question: overwrite or append on recompute?

When a window's identity changes (e.g. new prompt) or --force is passed: does the implementation overwrite the existing row, or append a new row and keep history?

Suspected: overwrite (one summary per window per current config). Concept §10.9 says --force "overwrites the 3 rows." The same should apply for a model/prompt change, unless a deliberate version history is wanted.

See [01] (coverage) and [03] (prompt_id).

Resolution

DecidedIdentity for idempotency = the window's capture set only.

  • A window counts as "done" when a preprocess row exists for that ordered capture set. model and prompt_id are stored as provenance, not part of the doneness check.
  • Changing the prompt or model does not auto-recompute. To pick up a new prompt/model, run with --force (recompute + overwrite the row; one summary per window). norm may warn when a row's stored prompt_id/model differs from current config, suggesting --force.
  • Consequence for REQ-CONFIG-003/004: verify the new prompt/model is used when a window is actually computed — a fresh/uncovered window or a --force re-run — not by a plain re-run over already-covered windows.

Rationale: the user chose "keep old summaries; regenerate only on --force," reversing this file's original suspicion. Keeps re-runs cheap and avoids silently re-spending the model when a prompt is tweaked.
