Extremely terse records of non-obvious decisions surfaced during implementation that don't belong in the concept doc, the requirements doc, or a commit message. Each refers out to durable ids (requirement ids, concept sections, commit hashes).

ADR-001 — Encrypted index: AES-256-GCM over serialized SQLite (not SQLCipher)

Affects: REQ-INIT-001 REQ-SEC-001 REQ-SEC-007 · concept §6, §7

Context. The concept/requirements named SQLCipher for the index. SQLCipher has no installable build for this target: no pip wheel for arm64/CPython 3.12 (only manylinux_x86_64), and it is not present via Homebrew. The realistic alternative was brew install sqlcipher + a source build of the sqlcipher3 binding on every dev/CI machine.

Decided The index is an ordinary SQLite database held in memory only while a command runs; at rest it is Connection.serialize() → AES-256-GCM → data_dir/index.db, and opened via the reverse (deserialize()). Blob and key-wrapping crypto are unchanged (AES-256-GCM blobs; Argon2id-wrapped data key).

Pure Python (cryptography + stdlib sqlite3), zero system dependencies. The security contract is satisfied unchanged: the on-disk file carries no SQLite format 3\0 header and is unreadable without the data key (REQ-SEC-001). Trade-off vs. SQLCipher's page-level encryption: the whole index is decrypted into process memory while unlocked — acceptable for norm's short-lived commands and its threat model, which already admits in-memory plaintext during operation, and the index holds metadata only (blobs stay out of it). Approved by the maintainer.

ADR-002 — Capture timestamps: local-naive ISO seconds, half-open range queries

Affects: REQ-DATA-002 REQ-GLOBAL-009 · concept §6, §10.13 · binds REQ-RECORD-001, REQ-DATA-006

Context. capture.ts is TEXT in the index (ADR-001). list (and later prune/export) filter by a --from/--to range. SQLite text comparison is lexicographic, so range filtering is only correct if every ts shares one fixed, sortable string form.

Decided Stored ts is local-naive ISO-8601 to seconds (e.g. 2026-06-06T12:00:00, no offset). Range bounds resolve against one command-start now (timerange.parse_range) and render to the same form via timerange.to_db_ts(); queries are half-open [start, end) — ts >= start AND ts < end. The recorder (REQ-RECORD-001) must write ts in exactly this form.

Lexicographic order matches chronological order for a single fixed-width local form, so filtering and ORDER BY ts need no date parsing in SQL. Half-open ranges compose without double-counting boundaries and align with prune --before (ts < cutoff, REQ-DATA-006). Trade-off: a DST/offset change can locally reorder rows across the transition — acceptable for a personal activity log; an explicit-offset ISO value passed on --from/--to is compared in its own wall-clock, not converted.

Refinement (report iteration). Stored ts stays second-resolution, but to_db_ts() renders range bounds at full sub-second precision rather than truncating. A live now bound carries microseconds, so it sorts after any capture stamped in the same second; this keeps the half-open [start, end) end inclusive of the current second, so --last / --to now still cover a capture recorded in the same second the report runs (otherwise report preprocess / interval would drop a just-recorded frame). Whole-second inputs (microsecond 0) render unchanged, so explicit ISO bounds and prune --before semantics are unaffected.

ADR-003 — argparse: global flags either side of the subcommand; leading-dash time values

Affects: REQ-GLOBAL-008 REQ-DATA-002 REQ-GLOBAL-009

Context. The contract wants --json both before (norm --json list, REQ-GLOBAL-008) and after (norm list --json, REQ-DATA-002) the subcommand, and relative time values like --from -1h (REQ-GLOBAL-009). Plain argparse fights both: a subparser flag with a normal default resets the value parsed before the subcommand, and a token beginning with - is taken as an unknown option, not a value.

Decided Each global flag is declared on the top-level parser (real defaults) and on every subparser with default=argparse.SUPPRESS, so an absent post-subcommand flag leaves the pre-subcommand value intact (cli._add_global_flags). Subparsers that take --from/--to/--last widen parser._negative_number_matcher to ^-\d+[smhdw]?$ (timerange.allow_relative_time_values) so -1h/-30d parse as values.

Both are the standard argparse idioms for these problems and keep the flag surface declared in one place. Safe because no norm subcommand has numeric or short-option flags that could collide with the widened matcher.

ADR-004 — AX "screen unchanged": concrete strip-list + coarse geometry buckets

Affects: REQ-RECORD-003 REQ-RECORD-004 · concept §8 ("Resolved details")

Context. Concept §8 says ax_hash is a normalized projection of the AX tree — keep structure/roles/labels/coarse geometry, strip focus, selection/caret, cursor, scroll offset, live clocks, ephemeral ids — and that "this strip-list defines screen unchanged." The concrete list is an implementation decision, and macapptree's exact AX JSON schema isn't pinned yet.

Decided hashing._normalize walks the AX JSON and, case-insensitively by key name: drops a fixed _STRIP_KEYS set (focus/selection/caret/cursor/scroll + id/identifier/uid/uuid/pid/window_id as "ephemeral ids"); rounds numeric geometry components (x,y,w,h,width,height,left,top,right,bottom, and list-valued position/size/frame/bounds/rect) to a 50 px grid; keeps everything else (roles, labels, nesting). ax_hash is SHA-256 over the canonical (sort_keys) JSON of that projection.

Schema-tolerant: it matches volatile keys wherever they appear rather than assuming macapptree's layout, so it survives the real binding landing later. Geometry bucketing absorbs sub-pixel layout jitter while a real move crosses a bucket. Deferred: "live clocks" need value-pattern heuristics, not key stripping — left until the real AX schema is integrated; the RECORD-003/004 contract (dedupe needs both phash and AX equal) holds regardless. The math is unit-tested below the CLI in tests/test_hashing.py.

ADR-005 — Idle gap buffered in the index meta table across ticks (max, not sum)

Affects: REQ-RECORD-002 · concept §8 · binds REQ-RECORD-001

Context. An idle tick stores nothing but must stamp the elapsed idle as idle_gap_s on the next stored capture (concept §8). record --once is a fresh process per tick, so the pending gap has to survive between invocations, and IOKit HIDIdleTime reports cumulative idle since the last input — successive idle reads overlap rather than add.

Decided The pending gap lives in the encrypted index meta table under pending_idle_s. An idle tick sets it to max(prior, observed); a stored capture reads it into idle_gap_s and resets it to 0. A deduped tick leaves it untouched.

max (not sum) avoids double-counting overlapping cumulative HIDIdleTime reads while guaranteeing idle_gap_s ≥ observed idle (the RECORD-002 contract). Reusing meta keeps the buffer inside the same atomic, encrypted flush as the capture write — no extra on-disk state. Scope: this iteration ships the seam-tested single-shot engine (--once); the persistent loop, signal handling, and launchd lifecycle (REQ-RECORD-005/006/009) and the real macapptree binding land next, mirroring how init deferred the model download.

ADR-006 — Recurring TDD chores live in a dev-only CLI outside the product, not as norm subcommands

Affects: REQ-GLOBAL-002 · conventions (verification.seam_note) · dev workflow only (no product req)

Context. The TDD loop kept re-deriving the same shell one-liners every session — a throwaway store for manual smoke runs (rm -rf /tmp/normsmoke; init; record …) and ad-hoc Python to pull a requirement out of norm-requirements.html — which then vanished into scrollback, recorded nowhere. They are real, repeated needs with no durable home. The product CLI surface can't absorb them: norm --help must list exactly the documented commands (REQ-GLOBAL-002), and "test seams are not product features."

Decided A developer-only package tools/normdev (run as python -m tools.normdev) exposes smoke and req. It lives outside src/ and is excluded from the wheel ([tool.hatch.build.targets.wheel] packages = ["src/norm"]), so it ships nothing and adds no norm subcommand. Its ephemeral-store driver (NormStore) is the single source of truth the pytest store fixture subclasses, so a manual smoke takes the same path as the black-box tests (pythonpath = ["."] lets the suite import it). req reads the doc's embedded const DATA object rather than scraping markup; "outstanding" is derived — a requirement is covered once its id (full REQ-X-NNN or short X-NNN) is referenced by a file under tests/. make wraps the no-arg cases.

Keeps the product CLI contract intact while giving the chores a place to be implemented, tested, and documented (CLAUDE.md "Developer tooling"). Sharing one driver means smoke and the acceptance tests can't drift. Reading the embedded data keeps the requirements doc hand-authored HTML (the source of truth) while making it queryable. Trade-off: coverage is a reference heuristic, not a real traceability matrix — naming a still-outstanding id in a test file would mark it covered; the dev-tooling tests avoid this by only citing already-covered ids. A future traceability test could harden this if it earns its keep.

ADR-007 — The record loop holds the data key in a scrubbable bytearray; zeroing is best-effort

Affects: REQ-RECORD-006 REQ-SEC-006 · concept §7

Context. Concept §7 holds that the unwrapped data key "lives only for the life of the command." For a one-shot command that life ends moments later when the process exits; the persistent record loop (REQ-RECORD-005) extends it indefinitely, so REQ-RECORD-006 requires the key be zeroed on shutdown — not merely left to process exit. But CPython offers no way to zero the immutable bytes that Argon2/AES-GCM return from unwrap_data_key.

Decided open_store still returns bytes; record converts it once to a bytearray working copy (dropping the original reference), uses that for every blob/index encrypt, and zeroes it via crypto.scrub() in a finally that covers the loop, the --once path, and error exits alike. SIGINT/SIGTERM set a stop event so the loop drains and that same finally runs — each tick already flushes the index, so the store is consistent and locked, the key is zeroed, and the process exits 0.

Scrubbing the single long-lived copy is the meaningful win for a process that may run for days; the transient immutable bytes from unwrap (and any copies the crypto backend makes internally) cannot be zeroed and are left to the GC plus process exit — an irreducible CPython limitation, not a norm gap, so the guarantee is honestly "best-effort." Threading a bytearray through session/store/blobs was rejected: churn across every reader for no added guarantee, since short commands exit immediately anyway. The loop is kept testable by two hidden seams (NORM_FAKE_MAX_TICKS to bound it, NORM_FAKE_SLEEP_LOG to make the cadence observable without waiting); the signal contract is asserted against a real subprocess.

ADR-008 — report interval: one planned-window set drives both coverage and aggregation; two prompts; declining the gap prompt aborts

Affects: REQ-INTERVAL-001 REQ-INTERVAL-003 REQ-INTERVAL-004 REQ-GLOBAL-010 · concept §10.10–10.12

Context. interval must answer two questions over a range: is it fully summarized? (to decide gap handling) and which preprocess summaries do we aggregate? A stored-row range query (list_preprocess(from,to), added speculatively in b5cf0a2) answers only the second, and the two answers can diverge — a preprocess row left over from a different --stride would be aggregated though it isn't part of the current window plan. Aggregation also needs a different prompt from the windows it may have to compute: a window summary is always a preprocess unit (prompt_preprocess), while the report itself uses prompt_interval.

Decided Both questions are answered from the same plan_windows(captures, window_k, stride_j) set: a window is "covered" iff preprocess_by_capture_ids finds its row, and the aggregation source is exactly those rows, in window order. So coverage and source can't disagree, and the unused list_preprocess was deleted (YAGNI; trivially restorable if a future inspector needs it). The gap-fill summarizes missing windows with a second _Settings(prompt_key="prompt_preprocess") while the command's --prompt overrides only the primary (prompt_interval) prompt; the model is loaded once and reused for both passes. Gap policy: --strict always aborts with COVERAGE_MISSING (exit 5) before any model load (so it outranks a MODEL_ERROR); otherwise --auto-preprocess or a non-interactive run (no TTY, or --json) takes the default and summarizes (REQ-GLOBAL-010); an interactive run prompts, defaulting to yes. A declined prompt aborts with the same COVERAGE_MISSING rather than emit a misleading partial report. inference.aggregate() is a text-only sibling of generate() (no images); the NORM_FAKE_MODEL trace gained n_summaries so a test can prove the interval pass consumed preprocess markdown, not raw captures.

Sharing one window set is the cheap invariant that keeps the dry-run preview, the coverage decision, and the stored source_preprocess_ids mutually consistent. The interactive prompt can't be reached by the black-box subprocess suite (its stdin is closed, so never a TTY), so the policy lives in a small pure _resolve_gap unit-tested directly (tests/test_interval_gap.py); the non-interactive branches are also asserted black-box. Trade-off: coverage/aggregation track the window_k/stride_j in effect at interval time, so preprocess rows made under a different stride are ignored rather than half-counted — the honest, predictable behaviour for a config the user controls.

ADR-009 — export/prune: one half-open range seam keyed per type; export's report predicate mirrors prune's

Affects: REQ-DATA-005 REQ-DATA-006 · concept §10.15–10.16

Context. Both commands select rows by time over the three artifact tables, but the spec pins the predicate for only one of them: §10.16 says prune matches captures on ts<cutoff, preprocess on window_end<cutoff, and interval reports on range_to<cutoff; §10.15 says only that export writes "data in range", naming no column. A summary/report has two natural time anchors (its window's start or end / its range's from or to), so "in range" is ambiguous for the report tables.

Decided One half-open [start, end) range selector, keyed on the per-table column prune already dictates — capture.ts, preprocess.window_end, interval_report.range_to — backs both commands (store._rows_in_range + the three typed wrappers). prune passes (None, cutoff) so < cutoff falls out as the same half-open form; export passes the parsed window. Export's report predicate therefore mirrors prune's (a summary belongs to a range by where it ends), so the two commands agree on which reports a range contains. The --include vocabulary (images,ax,reports; default all, reports = both summary kinds) is shared in norm.artifacts; prune restricts it to reports. prune removes index rows and flushes the encrypted index before unlinking any blob, so an interrupted run leaves at most a harmless orphan file — never a persisted row pointing at a missing blob (the inverse the contract forbids). Blob unlinking moved to blobs.delete_blob, shared with the --force preprocess overwrite.

Keying export and prune the same way means a norm export --from X --to Y followed by norm prune --before Y act on a consistent notion of "the reports in this range", and the row-selection math is unit-coverable in one place. Trade-off: a window straddling a bound is placed by its end alone, so an export range can omit a summary whose window opened before --from but closed inside it — acceptable, since a window's identity and stored bounds already treat its end as canonical, and prune relies on the same edge.

ADR-010 — passwd proves the current password from its own sources, not the unlock chain

Affects: REQ-SEC-005 · concept §10.18

Context. Every other data command resolves the app password through the unlock chain (NORM_PASSPHRASE > chmod-400 ~/.norm/passphrase file > interactive prompt), so the obvious move is to reuse it for the old password passwd needs. But rotation is the one operation whose whole point is to change the access gate: if it accepted the same headless sources, an unlocked daemon's environment or its stored passphrase file would silently authorize a rotation by anyone who reached that process — the current password would never actually be proven by a human.

Decided passwd reads the current password from NORM_OLD_PASSPHRASE or a no-echo prompt only — deliberately no NORM_PASSPHRASE and no passphrase-file fallback (acquire_old_passphrase, distinct from the unlock-path acquire_passphrase). The new password reuses the init confirmation path parameterized to NORM_NEW_PASSPHRASE (acquire_new_passphrase). The old password is then verified by actually unwrapping the data key (not a stored hash compare) before the new one is acquired, so a wrong current password fails (STORE_LOCKED, exit 3) before norm ever reads or prompts for a replacement. Rotation re-wraps that same data key under the new password and atomic_writes key.json; blobs and the index are not re-encrypted. The unwrapped key lives only in a scrubbed bytearray (per ADR-007) and nothing plaintext — neither key nor either password — is ever written.

The two non-interactive seams (NORM_OLD_PASSPHRASE + NORM_NEW_PASSPHRASE) keep scripted rotation possible without widening the authorization surface to the ambient unlock sources. Trade-off: a headless rotation must supply the current password explicitly rather than leaning on the daemon's existing unlock material — intentional friction, since changing the gate is exactly when "whoever can already unlock" is the wrong authority.

ADR-011 — Security-invariant guards assert on the source via ast, with a subprocess allowlist tripwire

Affects: REQ-SEC-002 · REQ-SEC-003 · REQ-SEC-006

Context. Three security requirements are negative-space invariants ("norm never reads the login password", "no Keychain item", "no elevated privileges"): there is no positive behavior to drive, so a black-box CLI test can't observe them directly. The tempting shortcut — grep the source for Keychain / LaunchDaemon — is wrong here: those very words appear (intentionally) in docstrings that document the absence, so a text scan flags the comments that promise the invariant holds. Equally, a runtime "did a Keychain item appear?" check would touch the real user's login keychain, making the suite environment-dependent and side-effectful.

Decided tests/test_security_invariants.py parses src/norm with ast and asserts on real code, not text: no forbidden import (keyring/pam/…), no subprocess/ os.system whose argv[0] is a Keychain or login-auth tool (security, dscl, sudo, …), and no os.set*id privilege call — so docstring prose is invisible to the guard. SEC-006 ("transient, in-memory") is asserted both black-box (preprocess writes no plaintext PNG/AX under the data dir or an isolated TMPDIR) and below-the-CLI (report._load_window returns in-memory PIL.Image objects and adds no file to the data dir). The static guards' teeth are verified against a planted-violation source tree, since they otherwise pass vacuously.

A deliberate second assertion narrows the observed shell-outs to an explicit allowlist (ioreg, launchctl) rather than only excluding the forbidden set. Trade-off: adding any new subprocess to norm fails this test until the program is registered — intentional friction that turns "norm grew a shell-out" into a conscious, reviewed change rather than a silent way past the Keychain/privilege guard.

ADR-012 — Model availability is checked inside load_model; a fake-cache seam decides present/absent

Affects: REQ-INIT-004 · REQ-PREPROCESS-005 · REQ-ARCH-001 · concept §10.1, §10.8

Context. norm init is the only command allowed to touch the network, and report must fail cleanly (MODEL_UNAVAILABLE, exit 4, naming norm init) when weights are absent rather than silently downloading them. Two questions follow: where does report assert availability, and how do the tests exercise "present" vs "absent" without the multi-GB weights or the network. A natural-looking spot is the report command layer, but the idempotent re-run (concept §10.9) loads no model at all — guarding there would force a weights check on a pure no-op.

Decided inference.ensure_available is called from inside inference.load_model, so the single load path is the single guard: the "0 new windows" re-run, which never loads, correctly needs no weights, while every real load (preprocess and interval gap-fill/aggregate) is checked before any row is written and before any download could happen. A malformed model_ref is rejected first by a dependency-free repo-id shape check (INVALID_MODEL_REF), keeping the 'invalid ref' vs 'not downloaded' distinction (PREPROCESS-005) sound even offline. Tests drive the cache through a hidden NORM_FAKE_MODEL_CACHE=<dir> seam: init "downloads" by dropping a models--org--name marker and ensure_available checks for it. The seam takes precedence over NORM_FAKE_MODEL, so a test can force the 'weights absent' path even with the generate() spy enabled. ARCH-001 is asserted by running a full report in-process under a socket.socket/create_connection guard that fails on any call.

huggingface_hub (download + offline cache probe) and mlx-vlm stay lazily imported, never hard dependencies — mirroring how the real backend is "finalized alongside" provisioning (the existing _load_backend pattern). Trade-off: the real provisioning/loading paths aren't exercised by the suite (the seams are), so they carry the same deferred-binding risk as the mlx-vlm backend; in return the test environment stays free of the heavy Apple-Silicon ML stack and every report test runs fast and deterministic.

ADR-013 — Declaring the deferred capture/inference deps: a pytest-pin override and a permission-probe seam gate

Affects: REQ-ENV-001 · REQ-RECORD-007 · REQ-ARCH-001 · REQ-INIT-004 · concept §8 · ADR-012

Context. The heavy runtime deps were deferred (ADR-012: mlx-vlm/huggingface_hub "stay lazily imported, never hard dependencies"; the macapptree and pyobjc bindings likewise unwired), so the project passed its suite through seams but could not actually init/record/report on a real machine. Declaring them surfaced two integration warts that the seam-only path had hidden.

Decided macapptree, mlx-vlm, huggingface-hub, and pyobjc (Quartz, ApplicationServices) are now declared runtime deps. The code still imports mlx-vlm/huggingface_hub/macapptree lazily (their absence still yields the contractual MODEL_UNAVAILABLE / macapptree-missing errors), but the heavy Apple-Silicon ML stack is now present in every environment — the trade-off ADR-012 had deferred.

pytest-pin override. macapptree 0.0.2 mis-declares pytest==7.4.4 as a runtime dependency (it is used only by its own test suite), colliding with our dev pytest>=8. pytest is never imported by macapptree at runtime, so [tool.uv] override-dependencies = ["pytest>=8"] forces one compatible version graph-wide instead of pinning the project back to 7.x.

Permission-probe seam gate. Installing pyobjc made capture._check_real_permissions actually run (CGPreflightScreenCaptureAccess / AXIsProcessTrusted); it had silently deferred while the frameworks weren't importable, so every fake-capture test broke in any environment lacking the OS grant. The real probe now runs only when a real capture will: it is skipped under either capture seam (NORM_FAKE_CAPTURE supplies frames, NORM_FORCE_NO_MACAPPTREE forces the backend absent for ENV-001), while NORM_FORCE_NO_PERMISSION still drives the denied path (RECORD-007). Trade-off: the genuine permission probe, like the real backends, is exercised only by a manual smoke run, not the suite.

ADR-014 — Real capture is in-process (Quartz + macapptree's UIElement), not macapptree's CLI

Affects: REQ-RECORD-001 · REQ-SEC-001 · REQ-ENV-001 · REQ-RECORD-007 · concept §8 · ADR-013

Context. With macapptree declared (ADR-013), _macapptree_frame had to become real. macapptree's public API (get_tree_screenshot) is unfit for a passive recorder: it activates the target app (steals focus) and sleeps, shells out to a bare python -m macapptree.main (which won't resolve to the venv interpreter under a launchd daemon), and writes the screenshot and AX JSON to temp files — plaintext capture on disk, which REQ-SEC-001 forbids.

Decided capture is assembled in-process from macapptree's lower-level pieces and Quartz, never its CLI. The frontmost app is read via NSWorkspace (never activated — no focus change); its main-window AX tree is built with macapptree's UIElement and serialized by to_dict() in memory; the screenshot is taken with CGDisplayCreateImage and encoded to PNG via NSBitmapImageRep, also in memory. No screencapture / python -m subprocess and no temp files, so no plaintext capture ever touches disk (SEC-001). The Frame image is decoded from the very bytes that get encrypted, so the stored blob and the dedupe image are identical.

Permission gate. The authoritative Screen-Recording check is the ensure_available preflight (CGPreflightScreenCaptureAccess), run before the store unlocks (RECORD-007, exit 6): without the grant CGDisplayCreateImage returns a black image rather than None, so it cannot self-detect a missing permission — the None check in the capture remains only as a defensive guard for access revoked mid-run.

Deferred: on multi-monitor setups the main display is always captured rather than the one holding the active window (tracked as REQ-RECORD-010, resolved in ADR-015); AX depth is unbounded (macapptree's default), relying on the downstream ax_hash normalization (ADR-004). Trade-off / risk: the native path can't be exercised deterministically — the suite unit-tests the composition glue with the three native probes stubbed, and a real capture is a single permission-gated smoke test (skipped where Screen Recording / Accessibility aren't granted), the same deferred-binding risk ADR-012 names for the mlx-vlm backend.

ADR-015 — Active-display capture: a pure window-center hit-test over a thin native display-enumeration seam

Affects: REQ-RECORD-010 · concept §8 · ADR-014

Context. ADR-014 shipped real capture but deferred multi-monitor: it shot CGMainDisplayID() unconditionally, so when the frontmost window lived on a secondary display the screenshot and the AX tree described different displays — exactly what REQ-RECORD-010 forbids. The native display APIs can't be exercised deterministically, the same bind-only-on-a-real-machine risk ADR-014/ADR-012 name, so the resolution had to keep the decision logic out of the un-testable native layer.

Decided the screenshot targets the display holding the active window. The window's on-screen rect is read from the same macapptree UIElement the AX tree is serialized from (absolute_position + size), so the shot and the tree provably describe one window — the REQ-RECORD-010 "same window/display" criterion holds by construction rather than by a second, possibly divergent, window lookup.

Pure selector over a thin seam. The choice is a pure function (_select_display): the display whose CGDisplayBounds rect contains the window's center, else a fallback. It is unit-tested deterministically with synthetic display rects (REQ-RECORD-010's pass/fail asserted with no real screen). The only native parts are thin, stubbable seams — _online_display_bounds (enumerate via CGGetOnlineDisplayList + CGDisplayBounds) and CGMainDisplayID — covered by a permission-gated smoke test that cross-checks every online display against the OS hit-test (CGGetDisplaysWithPoint).

Why center-containment, not CGGetDisplaysWithPoint directly: handing the point to the OS would push the whole decision into the un-testable native layer; the pure hit-test keeps it in the suite while approximating how macOS attributes a window to the display holding the bulk of it. It works because AX absolute_position and CGDisplayBounds share one global, top-left-origin coordinate space (main display at the origin), so window and display rects compare with no flip. Fallbacks → main display: unknown window bounds (no AX position/size) or a center over a gap between displays resolve to CGMainDisplayID() rather than guessing, and the single-monitor common case never enumerates displays. AX-depth remains the only standing ADR-014 deferral.
